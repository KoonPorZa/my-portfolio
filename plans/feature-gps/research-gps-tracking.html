01 · Mental model

ระบบ Live Location ทำงานเป็น pipeline 5 ชั้น

แอปอย่าง Messenger หรือ Google Maps ไม่ได้ “เปิด GPS แล้วโยนจุดขึ้นจอ” ตรง ๆ แต่เป็น pipeline ที่คุม permission, battery, network, server broadcast, viewer auth และสถานะ stale/expired ตลอดเวลา

1. Location provider

มือถือคำนวณตำแหน่งจาก GPS, Wi‑Fi, cell tower, sensor fusion และคืนค่า lat/lng/accuracy/speed/heading

2. Tracker client

แอปหรือเว็บขอ permission, watch ตำแหน่ง, filter ตามระยะ/เวลา, queue ถ้าเน็ตหลุด

3. Ingest API

รับ packet ผ่าน HTTPS/WebSocket, ตรวจ token/session, validate accuracy และบันทึก latest + history

4. Realtime fanout

broadcast update ไปยังผู้ดูผ่าน WebSocket, Supabase Realtime, Firebase listener หรือ SSE/polling fallback

5. Viewer map

อัปเดต marker/polyline, แสดงความแม่นยำ, เวลาอัปเดตล่าสุด, ETA และเตือนเมื่อข้อมูลเริ่ม stale

A

ข้อมูลที่ส่งแต่ละครั้ง

ควรส่งเท่าที่จำเป็น: lat, lng, accuracy_m, speed_mps, heading_deg, client_ts, seq, session_id

B

ไม่ควรส่งทุก event ดิบ

GPS jitter ได้ โดยเฉพาะในเมือง/ใต้หลังคา จึงควรกรองด้วย accuracy, minimum distance, minimum interval และ max stale interval

C

ผู้ดูไม่ได้ต้องการทุกจุด

คนรู้จักต้องรู้ว่า “ตอนนี้อยู่แถวไหน/ข้อมูลล่าสุดเมื่อไร/ยังเดินทางอยู่ไหม” มากกว่าเส้นทางละเอียดระดับ turn-by-turn

02 · Feasibility

ทำบนเว็บได้ แต่ต้องยอมรับ boundary ของ browser

Web Geolocation API รองรับ getCurrentPosition() และ watchPosition() บน HTTPS และต้องให้ผู้ใช้กดอนุญาตก่อน เหมาะกับ MVP ที่เปิดหน้า tracker ไว้ระหว่างเดินทาง แต่ไม่เทียบเท่า native app ที่ทำงานตอนล็อกจอได้

เหมาะกับ /trip-01 ตอนนี้
  • เปิดหน้า /trip-01 บนมือถือของคุณหลังปลดรหัส
  • กด “Start sharing” เพื่อเริ่มส่งตำแหน่ง
  • ขอ screen wake lock เพื่อกันจอดับเท่าที่ browser รองรับ
  • คนดูเปิดลิงก์ viewer แบบ token จำกัดเวลา
ไม่เหมาะ ถ้าต้องการระดับ Messenger/Google Maps เต็มรูปแบบ
  • ล็อกจอแล้วยังส่ง GPS ถี่ตลอดทาง
  • สลับไปใช้แอปอื่นนาน ๆ แล้ว tracker ยังทำงานแน่นอน
  • ต้องการ reliability สูงในเส้นทาง 1,000 กม. โดยไม่ต้องคอยดูจอ
  • ต้องการ OS background permission และ battery optimization แบบ native
หัวข้อ Web /trip-01 MVP Native app / Capacitor / React Native คำแนะนำสำหรับทริปนี้
ขอ permission Browser prompt ต่อ origin; ต้องเป็น HTTPS OS permission หลายระดับ เช่น while-in-use / always เริ่มจากเว็บพอ เพราะเป็น use case ส่วนตัว
อัปเดตตอนเปิดจอ ทำได้ดีถ้าหน้า visible และเน็ตนิ่ง ทำได้ดี ใช้ wake lock + UI เตือน “อย่าล็อกจอถ้าต้องการ live”
อัปเดตตอนล็อกจอ ไม่น่าเชื่อถือ และสเปคระบุว่าอัปเดตส่งให้เอกสาร visible/active ทำได้ด้วย background location แต่ต้องจัดการ policy/permission ถ้าจะใช้จริงจังทั้งวัน ให้พิจารณา native ในเฟส 2
ความเร็วทำ MVP เร็วสุด: 1–3 วันสำหรับ prototype ถ้ามี Supabase/Firebase อยู่แล้ว ช้ากว่า: ต้องทำ app, permission, build, test device เริ่มเว็บก่อนเพื่อ validate UX

03 · Recommended architecture

สถาปัตยกรรมที่เหมาะกับโปรเจ็คนี้

โปรเจ็คนี้เป็น Next.js App Router และหน้า app/trip-01/trip-client.tsx เป็น Client Component อยู่แล้ว จึงเหมาะจะเพิ่ม tracker UI เฉพาะ client และแยก backend ingest/viewer ออกเป็น API + realtime service

Repo fact: package.json ใช้ Next.js 16.2.9, React 19.2.4, ยังไม่มี dependency ด้าน map/realtime เพิ่มเติม
Repo fact: app/trip-01/page.tsx เป็น Server Component ที่ render Trip01Client; browser APIs ต้องอยู่ใน Client Component ตาม Next docs ที่อ่านจาก node_modules/next/dist/docs/...
Repo fact: Trip01Client มี password gate ฝั่ง client hardcoded เป็น 1942; ใช้กันคนทั่วไปดูแผนได้ระดับหนึ่ง แต่ไม่ควรใช้เป็น security boundary สำหรับ live location
1

MVP แนะนำ: Supabase Realtime + Postgres

เหมาะกับโปรเจ็คนี้เพราะคุณคุ้นกับ Supabase/Firebase ใน stack เดิม และได้ทั้ง DB, RLS, realtime subscription, retention job ภายหลัง

  • Tracker POST ตำแหน่งผ่าน Next Route Handler หรือ Supabase function
  • Viewer subscribe ตาราง trip_location_latest ด้วย Postgres Changes หรือ Broadcast
  • ถ้าอัปเดตทุก 5–10 นาที ใช้ polling ทุก 30–60 วินาทีได้; realtime เป็น nice-to-have ไม่ใช่ requirement
2

ทางเลือกง่ายสุด: Firebase Realtime Database / Firestore

ดีมากสำหรับ listener realtime และ MVP ที่อยากลด backend code; ต้องระวัง security rules และค่าอ่าน/เขียนถ้าเปิด listener กว้างเกินไป

  • เขียน latest location ที่ path/session document
  • Viewer ใช้ onValue() หรือ onSnapshot()
  • ทำ TTL/cleanup และ auth rules ให้ชัด
Component หน้าที่ ตำแหน่งใน Next.js หมายเหตุ
Tracker panel Start/Stop sharing, permission status, last upload, wake lock status app/trip-01/trip-client.tsx หรือแยก live-tracker.tsx ต้องเป็น "use client" เพราะใช้ navigator.geolocation, window, state/effect
Ingest API รับตำแหน่งจาก owner, validate token, rate limit, upsert latest, insert history app/api/trips/trip-01/location/route.ts อย่า expose service role key ใน browser; ใช้ server-side secret เท่านั้น
Viewer route แผนที่สำหรับคนรู้จัก เปิดจาก signed link app/trip-01/live/page.tsx + client map ไม่ควรใช้ password gate เดิม; ใช้ viewer token ที่ revoke/expire ได้
Realtime ส่ง latest location ไป viewer Supabase channel / Firebase listener / WebSocket / SSE ถ้า deploy บน serverless ทั่วไป WebSocket custom อาจยุ่งกว่า managed realtime
Map layer แสดง marker, accuracy circle, polyline, PTT stops Leaflet/MapLibre/Google Maps client component Leaflet เริ่มง่าย; Google Maps ต้อง API key/billing และ mapId ถ้าใช้ Advanced Markers

04 · Cadence

“Realtime” กับ “ทุก 5–10 นาที” เป็นคนละ UX

แก้จากความเข้าใจเดิม: requirement ที่ถูกคืออัปเดตทุก 5–10 นาที ไม่ใช่เมตร ดังนั้น Web MVP น่าใช้ขึ้นมาก เพราะจำนวน point ทั้งทริปต่ำ แบต/เน็ตเบากว่า และไม่จำเป็นต้องใช้ realtime infrastructure ตลอดเวลา

5

5 นาที = จุดค่อนข้างสด

ที่ 90 กม./ชม. รถไปได้ประมาณ 7.5 กม. ต่อ 5 นาที เหมาะให้คนรู้จักรู้ว่า “อยู่แถวไหนแล้ว” โดยไม่ต้องเห็นทุกโค้ง

10

10 นาที = ประหยัดขึ้น

ที่ 90 กม./ชม. ระยะห่างระหว่างจุดประมาณ 15 กม. เหมาะช่วงถนนยาว/แบตต่ำ/เน็ตไม่ดี แต่ควรแสดง timestamp ชัดเจน

UX

Viewer ต้องเห็นความสด

ถ้าตำแหน่งล่าสุดเกิน 15 นาที ให้ขึ้น “ข้อมูลเริ่มเก่า”; เกิน 30 นาทีให้ขึ้น “offline / ไม่ได้อัปเดต” เพื่อไม่ให้เข้าใจผิดว่าเป็น realtime

ผลจาก research เพิ่ม: Android docs แนะนำให้ใช้ interval ใหญ่ที่สุดเท่าที่ use case รับได้ เพราะความถี่สูงใช้แบตมากขึ้น; สำหรับ foreground ที่ไม่ใช่ realtime แนะนำระดับ “ไม่กี่นาที” และเอกสาร geofencing ยังยกตัวอย่าง responsiveness 5–10 นาทีได้ ดังนั้น cadence 5–10 นาทีเข้ากับ use case “แชร์ตำแหน่งคร่าว ๆ ให้คนรู้จักดูระหว่างทริป” มากกว่า cadence รายวินาที
โหมด เงื่อนไขส่งตำแหน่ง เหมาะกับ ข้อดี/ข้อเสีย
Trip periodic · แนะนำ ทุก 5 นาที ขณะกำลังเดินทาง + ส่งทันทีเมื่อกด Start/Stop/Manual ping ทริปสงขลา → กรุงเทพฯ ที่อยากให้คนดูรู้ตำแหน่งล่าสุด สมดุลดี: ทริป 15 ชม. มีราว 180 จุด ไม่หนัก DB/แบต/เน็ต
Battery saver ทุก 10 นาที หรือเมื่อแบตต่ำ/มือถือร้อน/เน็ตไม่ดี ช่วงถนนยาว วิ่งคงที่ หรือเมื่อ power bank ไม่พร้อม ประหยัดขึ้น แต่ marker กระโดด 10–15 กม. ได้
Approach / city ทุก 2–3 นาที เฉพาะช่วงเข้าเมือง/ใกล้ waypoint สำคัญ ใกล้กรุงเทพฯ, พระราม 2, จุดนัดพบ, เลี้ยวเยอะ สดขึ้นช่วงจำเป็น แต่ไม่ควรเปิดทั้งวันถ้าไม่จำเป็น
Rest / stopped ส่งเมื่อถึงจุดพัก + ทุก 10–15 นาทีระหว่างพัก + ส่งทันทีเมื่อออกจากพัก จอดปั๊ม/กินข้าว/พักนาน ผู้ดูรู้ว่ายังปลอดภัยและจอดอยู่ โดยไม่เปลืองแบต
Realtime turn-by-turn ทุกไม่กี่วินาทีหรือเมื่อขยับหลายสิบเมตร navigation สดจริง, dispatch, delivery ไม่ใช่ requirement ตอนนี้; ต้องระวังแบตและ background reliability

ตัวอย่าง algorithm ฝั่ง tracker สำหรับทุก 5–10 นาที

ใช้ periodic snapshot ด้วย getCurrentPosition() จะตรงกับ requirement กว่า realtime watch ตลอดเวลา; ถ้าต้องการ marker สดขึ้นระหว่างเปิดจอ ค่อยใช้ watchPosition() เป็น optional mode แล้วจำกัดการ upload 

const ACTIVE_INTERVAL_MS = 5 * 60_000;   // เดินทางปกติ: ทุก 5 นาที
const SAVER_INTERVAL_MS = 10 * 60_000;   // ประหยัดแบต: ทุก 10 นาที
const REST_INTERVAL_MS = 15 * 60_000;    // พักนาน: ทุก 15 นาที
const STALE_AFTER_MS = 15 * 60_000;      // viewer เริ่มเตือนว่าข้อมูลเก่า
const OFFLINE_AFTER_MS = 30 * 60_000;    // viewer แสดง offline / ไม่อัปเดต
const MAX_BAD_ACCURACY_M = 250;

let seq = 0;
let timer = null;
let mode = "active"; // "active" | "saver" | "rest"

function intervalForMode() {
  if (mode === "saver") return SAVER_INTERVAL_MS;
  if (mode === "rest") return REST_INTERVAL_MS;
  return ACTIVE_INTERVAL_MS;
}

function scheduleNext() {
  clearTimeout(timer);
  timer = setTimeout(captureAndUpload, intervalForMode());
}

function captureAndUpload(reason = "scheduled") {
  navigator.geolocation.getCurrentPosition(async (pos) => {
    const point = {
      seq: ++seq,
      lat: pos.coords.latitude,
      lng: pos.coords.longitude,
      accuracyM: pos.coords.accuracy,
      speedMps: pos.coords.speed,
      headingDeg: pos.coords.heading,
      clientTs: new Date(pos.timestamp).toISOString(),
      reason,
      mode,
    };

    if (point.accuracyM <= MAX_BAD_ACCURACY_M || reason === "manual") {
      await uploadLocation(point);
    }

    scheduleNext();
  }, (error) => {
    reportTrackerError(error);
    scheduleNext();
  }, {
    enableHighAccuracy: mode !== "saver",
    timeout: 20_000,
    maximumAge: 60_000,
  });
}

// Start: ส่งทันที แล้วค่อยวนทุก 5–10 นาที
captureAndUpload("start");

// Manual ping: ปุ่มให้ rider กดส่งจุดล่าสุดทันทีเมื่อถึงปั๊ม/เกิดเหตุ
function manualPing() {
  captureAndUpload("manual");
}

05 · Privacy & security

Live location เป็นข้อมูลอ่อนไหว ต้องออกแบบให้ปิดง่ายและหมดอายุเอง

หน้านี้ตั้งใจให้ “คนรู้จัก” ดูเท่านั้น ดังนั้นห้ามใช้ public endpoint แบบไม่มี token และห้ามเก็บ route history ถาวรโดยไม่จำเป็น

ห้ามใช้ password ฝั่ง client เป็น security จริง

รหัสใน trip-client.tsx ถูก bundle ไป browser ได้ คนที่ inspect code อาจเห็นได้ จึงเหมาะแค่กันคน casual ไม่ใช่กัน live location

ใช้ signed viewer link

สร้าง URL เช่น /trip-01/live?t=... ที่ผูก session, expire, revoke ได้ และเก็บ hash ของ token ใน DB ไม่เก็บ token plain

มีปุ่ม Stop ชัดเจน

Owner ต้องหยุดแชร์ได้ทันที และระบบควรตั้ง TTL เช่น ปิด session อัตโนมัติหลัง 24 ชั่วโมงหรือเมื่อถึงปลายทาง

มาตรการ รายละเอียด เหตุผล
HTTPS เท่านั้น Geolocation และ wake lock ต้อง secure context; production deploy เช่น Vercel ใช้ HTTPS ป้องกันข้อมูลตำแหน่งหลุดระหว่างทาง และตรงตาม browser requirement
Token แยก owner/viewer Owner token ส่ง location ได้; viewer token อ่านได้อย่างเดียว ลด damage ถ้าลิงก์ viewer ถูก forward
RLS / API validation Supabase เปิด Row Level Security หรือผ่าน API server ที่ตรวจ session ทุกครั้ง กัน client เขียน/อ่านข้าม session
Retention ลบ history point หลัง 24–72 ชม.; เก็บ latest เมื่อ session active เท่านั้น ลดความเสี่ยงจากข้อมูลย้อนหลัง
Coarse mode optional ถ้าใกล้บ้าน/ที่พัก อาจ blur ตำแหน่งเป็น radius 200–500 เมตร หรือหยุดแชร์ ลดการเปิดเผยจุดพัก/บ้านแบบละเอียด
Stale state ถ้า update ล่าสุดเกิน 2–5 นาที แสดงสีเหลือง/แดงและข้อความ “ข้อมูลเก่า” ผู้ดูจะไม่เข้าใจผิดว่าตำแหน่งล่าสุดคือ realtime

06 · Implementation plan

แผนทำจริงแบบเป็นเฟส

ถ้าจะลงมือ implement ต่อ แนะนำแบ่งเป็น 3 เฟสเพื่อให้ได้ของใช้ได้เร็วและไม่ over-engineer ตั้งแต่แรก

Phase 1 — Local prototype บน /trip-01

  • เพิ่ม panel “Live Location” ในหน้า trip หลังปลดรหัส
  • ใช้ navigator.geolocation.getCurrentPosition() ตามรอบ 5–10 นาที หรือ watchPosition() เฉพาะตอนทดสอบความถี่สูง แสดงตำแหน่ง/accuracy ล่าสุดในหน้าเดียว
  • เพิ่ม Haversine filter, permission status, error state และปุ่ม Start/Stop
  • ยังไม่ส่งออก internet; ใช้ทดสอบบนมือถือจริงก่อน

Phase 2 — Backend + viewer link

  • เลือก Supabase หรือ Firebase
  • สร้าง session + viewer token + latest table/document
  • Route Handler สำหรับ ingest: validate token, rate limit, upsert latest, append history
  • หน้า /trip-01/live แสดงแผนที่ + marker + stale state + ETA ไป waypoint ถัดไป

Phase 3 — Reliability สำหรับเดินทางจริง

  • offline queue เมื่อเน็ตหลุด และ flush เมื่อกลับมาออนไลน์
  • wake lock opt-in + เตือนเมื่อ tab hidden / จอดับ / permission denied
  • retention cleanup, revoke link, audit log ว่าใครเข้าดู
  • fallback polling เมื่อ realtime subscribe หลุด

Phase 4 — Native ถ้าต้องการระดับ Messenger

  • ทำ React Native/Capacitor app หรือ companion app เฉพาะ tracker
  • ใช้ OS background location + foreground notification/indicator ตาม policy
  • ยังใช้ viewer web ของเดิมได้

trip_share_sessions

เก็บ session, trip_id, active, expires_at, owner_token_hash, viewer_token_hash, revoked_at

trip_location_latest

เก็บจุดล่าสุด 1 row ต่อ session เพื่ออ่านเร็วและ subscribe ง่าย

trip_location_points

เก็บ history แบบ append-only สำหรับวาดเส้นทาง ย้อนหลัง และ debug แต่ลบตาม TTL

trip_view_events

MDN — Geolocation API

ยืนยันว่า Geolocation API ต้องใช้ HTTPS, ต้องขอ permission และมี getCurrentPosition()/watchPosition() สำหรับตำแหน่งปัจจุบันและการติดตามการเปลี่ยนตำแหน่ง

https://developer.mozilla.org/en-US/docs/Web/API/Geolocation_API

MDN — getCurrentPosition() / watchPosition()

หลังแก้ requirement เป็นทุก 5–10 นาที ใช้ getCurrentPosition() แบบ periodic snapshot เป็น default ได้; watchPosition() ยังใช้ได้สำหรับ optional live mode และทั้งคู่รองรับ options เช่น enableHighAccuracy, timeout, maximumAge

getCurrentPosition() · watchPosition()

W3C Geolocation Candidate Recommendation

ระบุรายละเอียด PositionOptions, accuracy/heading/speed, privacy guidance และข้อจำกัดว่า position updates ถูกส่งให้เอกสารที่ fully active และ visible

https://www.w3.org/TR/geolocation/

MDN — WebSocket / Server-Sent Events

ใช้เปรียบเทียบ transport สำหรับ realtime fanout: WebSocket เป็นสองทาง ส่วน SSE เป็น server-to-client ทางเดียว เหมาะเป็น fallback หรือ viewer-only stream

https://developer.mozilla.org/en-US/docs/Web/API/WebSocket · SSE guide

Supabase Realtime Docs

ยืนยันการใช้ Broadcast ผ่าน WebSocket และการ subscribe Postgres Changes พร้อม RLS/replication setup; เหมาะกับ MVP ที่ต้องการ DB + realtime

https://supabase.com/docs/guides/realtime/broadcast · Postgres Changes

Firebase Realtime Database / Firestore Docs

ใช้เป็นทางเลือก managed realtime: Realtime Database มี listener ที่ trigger เมื่อข้อมูลเปลี่ยน; Firestore มี onSnapshot() สำหรับ realtime document updates

https://firebase.google.com/docs/database/web/read-and-write · Firestore listen

Google Maps JavaScript API / Leaflet

ใช้เทียบ map renderer: Google Maps Advanced Markers ต้องโหลด marker library และใช้ API key; Leaflet เป็น open-source mobile-friendly interactive map และใช้ marker/polyline ได้ง่าย

Google Maps Advanced Markers · Leaflet Quick Start

Android Developers — Background location

ใช้ยืนยันว่าการทำ background location แบบ native ต้องขอ permission เฉพาะ, ต้องเป็น core functionality และ Android จำกัด background updates เพื่อประหยัดแบต

https://developer.android.com/develop/sensors-and-location/location/background

Android Developers — Battery & real-world location scenarios

Battery guide · Real-world scenarios
