01 · Target outcome

ผลลัพธ์ที่ต้องได้

ฟีเจอร์นี้ไม่ใช่ navigation app เต็มรูปแบบ แต่เป็น live-trip status สำหรับให้คนรู้จักดูว่าตอนนี้ rider อยู่แถวไหนแล้วระหว่างเดินทางในหน้า /trip-01

1

Owner tracker บน /trip-01

หลังปลดรหัสหน้า trip แล้ว rider เห็น panel “Live Location” พร้อม Start/Stop, mode, last sent, accuracy, permission และ manual ping

2

Viewer page สำหรับคนรู้จัก

คนดูเปิดลิงก์ viewer แล้วเห็นตำแหน่งล่าสุด, เวลาอัปเดต, stale/offline state, route stops และ Google Maps link หรือ map marker

3

Privacy-first sharing

แชร์ด้วย token ที่หมดอายุ/revoke ได้, owner หยุดแชร์ได้ทันที, ไม่ใช้รหัส client-side เดิมเป็น security boundary สำหรับข้อมูลตำแหน่งจริง

02 · Branch & preflight

เริ่มงานบน branch feat/gps เท่านั้น

สถานะ repo ตอนวางแผน: branch ปัจจุบันคือ develop และมีไฟล์ modified/untracked หลายรายการอยู่แล้ว ดังนั้นก่อนลงมือ implement ต้องระวังไม่ให้ทับงานเดิม แผนนี้กำหนด branch เป้าหมายเป็น feat/gps แต่ยังไม่ได้ switch branch ให้โดยอัตโนมัติ
Preflight ที่ต้องทำก่อนเขียน code: ตรวจ git status --short, ยืนยันว่าไฟล์ที่กำลังจะเปลี่ยนเป็น scope ฟีเจอร์ GPS, แล้วค่อย git switch -c feat/gps หรือ git switch feat/gps
Step Command / action Pass condition ถ้าไม่ผ่าน
1. Check current branch git branch --show-current เห็น feat/gps ก่อนเริ่มแก้ code switch/create branch ก่อน
2. Check dirty worktree git status --short เข้าใจว่าไฟล์ไหนเป็นงานเดิม ไฟล์ไหนเป็นงาน GPS อย่า reset/delete; แยก commit หรือจดรายการก่อน
3. Install dependency gate ยังไม่เพิ่ม map/realtime dependency จนเลือก provider ไม่มี dependency ใหม่โดยไม่จำเป็น ใช้ Google Maps link/plain coordinates เป็น MVP fallback
4. Env gate เตรียม .env.local เฉพาะเครื่อง secret ไม่ถูก commit และ server-only key ไม่เข้า client bundle หยุด integration แล้วใช้ mock/local mode ไปก่อน

03 · Architecture decision

สถาปัตยกรรมแนะนำสำหรับ MVP

ADR-001

ใช้ Web MVP + periodic upload เป็น default

จาก research: browser ทำ GPS ได้ดีตอนหน้า active/visible แต่ไม่ควรรับประกัน background tracking ทั้งวัน ดังนั้น implementation รอบแรกควรทำให้ใช้งานได้เมื่อเปิดหน้าไว้ และแจ้งข้อจำกัดชัดเจน แทนที่จะพยายามเลียนแบบ native app ตั้งแต่แรก

  • Chosen: getCurrentPosition() ทุก 5 นาที, saver mode ทุก 10 นาที, manual ping
  • Transport: HTTPS POST ไป Route Handler; viewer ใช้ polling 30–60 วินาที หรือ realtime provider ภายหลัง
  • Backend preference: Supabase Postgres ผ่าน server route; ถ้า credential ยังไม่พร้อมให้ทำ mock adapter ก่อน
  • Map: เริ่มด้วย coordinates + Google Maps link; เพิ่ม vanilla Leaflet เป็น enhancement ไม่ใช้ React Leaflet ใน MVP
Non-goals

สิ่งที่ยังไม่ทำใน MVP

  • ไม่รับประกันการส่งตำแหน่งตอนล็อกจอหรือปิด browser
  • ไม่ทำ turn-by-turn realtime ระดับทุกวินาที
  • ไม่เปิด public endpoint แบบไม่มี token
  • ไม่เก็บ route history ถาวรเกิน retention ที่กำหนด
  • ไม่เพิ่ม dependency/map SDK โดยไม่ผ่าน decision gate

04 · Tech stack decision

เคาะ tech stack สำหรับ GPS MVP

เลือก stack ที่เพิ่ม dependency น้อยที่สุด ใช้ของที่ repo มีอยู่แล้วให้มากที่สุด และยังเหลือทางอัปเกรดเป็น realtime/native ได้ถ้าทดสอบจริงแล้ว Web MVP ไม่พอ

Decision: ใช้ Next.js App Router + Route Handlers + Supabase Postgres + periodic polling + vanilla Leaflet optional. ไม่ใช้ custom WebSocket, ไม่ใช้ Firebase ในรอบแรก, และไม่ทำ native app จนกว่าจะพิสูจน์ว่า browser background limitation เป็น blocker จริง
Layer Chosen stack เหตุผล Fallback / future
Frontend app Existing Next.js 16 App Router, React 19, TypeScript, CSS Modules ตรงกับ repo ปัจจุบัน ไม่เพิ่ม framework/UI library ใหม่ และหน้า /trip-01 เป็น Client Component อยู่แล้ว ถ้า UI ใหญ่ขึ้นค่อยแยก design system; ยังไม่ต้องใช้ shadcn/MUI
GPS capture Browser navigator.geolocation.getCurrentPosition() แบบ periodic snapshot ตรง requirement ทุก 5–10 นาทีมากกว่า watchPosition() ตลอดเวลา ลดแบต/เน็ต และทำงานง่ายกว่าใน browser watchPosition() เป็น optional live/debug mode; native app เป็น future phase ถ้าต้อง track ตอน lock screen
Wake / visibility UX Screen Wake Lock API optional + Page Visibility warning ช่วยให้ owner รู้ว่า browser อาจหยุดอัปเดตเมื่อจอดับ/แท็บ hidden โดยไม่ claim เกินจริง ถ้า Wake Lock ไม่รองรับ ให้แสดง warning และใช้ manual ping
API boundary Next Route Handlers: app/api/trips/trip-01/location/route.ts ซ่อน server secrets, validate token/payload server-side, ใช้ runtime เดียวกับแอปเดิม ถ้าต้องรองรับหลาย trip ค่อย generalize เป็น /api/trips/[tripId]/location
Runtime Node.js runtime สำหรับ route handler ใช้ crypto, server-only env และ Supabase server client ได้ชัดเจนกว่า Edge ใน MVP Edge runtime ยังไม่จำเป็น เพราะ update ทุก 5–10 นาที ไม่ latency-sensitive
Database Supabase Postgres ผ่าน @supabase/supabase-js@2 ฝั่ง server เท่านั้น เหมาะกับ session/latest/history schema, ทำ TTL/retention ได้ง่าย, และต่อยอด Realtime ได้ถ้าจำเป็น Firebase เป็น backup ถ้าต้องการ managed realtime listener ง่ายสุด แต่ไม่เลือกใน MVP เพราะ polling พอแล้ว
Realtime strategy Viewer polling ทุก 30–60 วินาที GPS update จริงทุก 5–10 นาที จึงไม่คุ้มเพิ่ม WebSocket/Realtimes ตั้งแต่แรก Supabase Realtime/Postgres Changes เป็น P2 ถ้าคนดูหลายคนหรือ UX ต้องสดขึ้น
Map renderer MVP: Google Maps external link + coordinates. Enhancement: vanilla leaflet client-only map เริ่มได้ทันทีโดยไม่มี key/billing; Leaflet เพิ่ม map marker/accuracy circle ได้โดยไม่ผูกกับ React wrapper ใช้ OSM tiles เฉพาะ low-traffic พร้อม attribution; ถ้า public/high-traffic ให้เปลี่ยน tile provider เช่น MapTiler/Mapbox/Google Maps
Tile provider Configurable tile URL; default ใช้ OSM standard tiles เฉพาะส่วนตัว/traffic ต่ำ OSM tile servers มี policy ชัดเจนเรื่อง attribution, referer, caching และไม่รับประกัน SLA จึงต้องไม่ hard-code แบบเปลี่ยนไม่ได้ ตั้ง env/config เช่น NEXT_PUBLIC_TRIP_MAP_TILE_URL เพื่อเปลี่ยน provider ได้
Validation Manual validation utilities ใน lib/trip-gps/validation.ts payload เล็กมาก: lat/lng/accuracy/timestamps/mode/reason; ยังไม่จำเป็นต้องเพิ่ม zod ถ้า API โตขึ้นหรือมีหลาย payload ค่อยเพิ่ม zod
Token/security Node crypto: random token + SHA-256 hash stored server-side ไม่ต้องเพิ่ม auth package; token แยก owner upload กับ viewer read-only, expire/revoke ได้ ถ้าต้องมี user login จริง ค่อยใช้ Supabase Auth ภายหลัง
Owner start auth Server-side live-share code: owner กรอก code เฉพาะ GPS sharing เพื่อ create session ไม่ reuse password เดิมที่อยู่ client-side เป็น security boundary สำหรับ live location อนาคตเปลี่ยนเป็น Supabase Auth หรือ passkey ได้
Testing Existing npm run lint + npm run build + manual mobile HTTPS test repo ยังไม่มี test runner; MVP ไม่ควรเพิ่ม test dependency ก่อนมี domain logic เยอะ ถ้าเพิ่ม unit tests ให้เลือก vitest เป็น P2
Dependencies to add

เพิ่มเท่าที่จำเป็น

  • @supabase/supabase-js — ใช้เฉพาะ server route/store adapter
  • leaflet — เพิ่มเฉพาะเมื่อทำ map enhancement; MVP แรกใช้ Google Maps link ได้ก่อน
Dependencies not now

ยังไม่เพิ่มใน MVP

  • react-leaflet — เลี่ยง compatibility/SSR complexity; ใช้ vanilla Leaflet ใน client component
  • socket.io / custom WebSocket — polling พอสำหรับ 5–10 นาที
  • firebase — ยังไม่เลือก เพราะ Supabase Postgres เหมาะกับ session/security/history มากกว่า
  • zod / vitest — optional future ถ้า feature โตขึ้น

05 · Cost guard

Cost expectation: MVP ควรเป็น 0 บาท ถ้าอยู่ใน free quota

Stack ที่เคาะไว้ตั้งใจเลี่ยง paid API: ไม่ใช้ Google Maps JavaScript API, ไม่ใช้ custom realtime, ไม่ใช้ paid tile provider และใช้ Supabase/Vercel free tier เท่าที่พอสำหรับทริปส่วนตัว

06 · Cloudflare edge layer

Cloudflare services ที่เลือกใช้: DNS/Proxy/CDN, WAF Rate Limiting, Web Analytics

Cloudflare ในแผนนี้เป็นชั้นหน้าเว็บ ไม่ใช่ backend server เพิ่ม: ให้ request ผ่าน Cloudflare ก่อนถึง Vercel เพื่อเปิด security, rate-limit และ analytics แบบไม่เปลี่ยน Next.js full-stack architecture เดิม

Service ใช้ทำอะไรในโปรเจ็คนี้ Setup ในแผน Guardrail
1. DNS / Proxy / CDN ให้ domain เช่น koonporza.com วิ่งผ่าน Cloudflare orange-cloud proxy ก่อนถึง Vercel เพื่อใช้ Cloudflare protection, cache/static optimization, HTTP analytics และ WAF rules
  • ตั้ง DNS record ของเว็บเป็น Proxied ถ้า record นั้น serve HTTP/HTTPS traffic
  • คง Vercel verification/TXT หรือ record ที่ไม่ใช่ web traffic เป็น DNS-only ตามที่ provider ต้องการ
  • ตั้ง SSL/TLS เป็น Full/Strict เมื่อ certificate ฝั่ง origin พร้อม
  • ห้าม cache API location; cache ได้เฉพาะ static assets/page ที่ปลอดภัย
 อย่าเปิด cache rule ครอบ /api/trips/trip-01/* หรือ /trip-01/live*; location response ต้องเป็น dynamic และไม่ cache
2. WAF + Rate Limiting กัน bot/spam/brute-force ที่ยิง location API หรือ viewer token endpoint ก่อน request ถึง Vercel/Supabase
  • Custom WAF rule scope: /api/trips/trip-01/location*, session endpoints, และ optional /trip-01/live*
  • Rate limit POST owner upload สูงกว่าปกติไม่ได้ เช่นเกินประมาณ 10–20 request/min/IP ให้ challenge/block
  • Rate limit GET viewer latest ให้สูงกว่าการ poll ปกติ เช่น 30–60 request/min/IP เพื่อไม่กระทบ viewer polling 30–60 วินาที
  • ยังต้องมี server-side token validation/rate limit ใน Next Route Handler เสมอ
 WAF เป็น defense-in-depth ไม่ใช่ auth; ถ้า Cloudflare rule พลาด API route ต้องยัง reject invalid token/payload ได้เอง
6. Web Analytics ดู traffic/performance แบบ aggregate ของ portfolio, /trip-01, และหน้า public ที่ไม่ sensitive
  • เพิ่ม Cloudflare Web Analytics beacon เฉพาะ layout/page ที่ไม่ส่ง token ใน URL
  • ใช้ดู page views, device/browser, performance trend หลัง deploy
  • ถ้าจะ track viewer page ให้ทำเฉพาะแบบ sanitized route ที่ไม่มี token/query หรือใช้ internal view counter แทน
 ห้ามฝัง analytics บน URL ที่มี viewer token เช่น /trip-01/live?t=...; token ไม่ควรถูกส่งเข้า analytics/log ภายนอก
Implementation note

ยังไม่ต้อง deploy backend ใหม่

Cloudflare layer นี้อยู่หน้า Vercel เท่านั้น: Next Route Handlers ยังเป็น backend API เดิม, Supabase ยังเป็น database เดิม, และ branch implementation ยังคือ feat/gps

Sensitive URL rule

อย่าให้ token อยู่ใน analytics

ถ้า viewer token อยู่ใน query/path ให้ปิด analytics beacon บน viewer page หรือเปลี่ยนเป็น flow ที่ไม่ expose token ใน URL analytics ก่อนเปิด Web Analytics กับหน้านั้น

07 · Viewer & owner GPS flow

รายละเอียดที่ต้องทำสำหรับคนดู และส่วนที่คุณต้องเปิด GPS

ฟีเจอร์นี้มี 2 ฝั่งที่ต้องออกแบบให้ชัด: ฝั่ง viewer ที่เปิดดูแบบ read-only และฝั่ง rider/owner ที่ต้องเปิด GPS + กดเริ่มแชร์เองก่อนตำแหน่งจะถูกส่งออกไป

Viewer detail

หน้าที่ของหน้า /trip-01/live

Viewer page คือหน้าสำหรับคนรู้จักที่ได้รับลิงก์จากคุณ เปิดดูตำแหน่งล่าสุดได้อย่างเดียว ไม่สามารถสั่ง start/stop หรือเขียนตำแหน่งแทนคุณได้

  • Entry URL: /trip-01/live?t=<viewer-token> หรือ path token ที่เทียบเท่า
  • Auth: ส่ง token ไปตรวจที่ API; token valid + session active เท่านั้นถึงอ่าน latest location ได้
  • Data refresh: polling GET /api/trips/trip-01/location ทุก 30–60 วินาที เพราะ GPS update จริงทุก 5–10 นาที
  • Map fallback: MVP แสดงพิกัด + ปุ่ม “เปิดใน Google Maps” ก่อน; map marker/accuracy circle เป็น enhancement ภายหลัง
  • Read-only safety: viewer token ต้อง POST ไม่ได้ และไม่เห็น owner token หรือข้อมูลที่ใช้ควบคุม session
Owner GPS detail

สิ่งที่คุณต้องทำเพื่อเปิด GPS และเริ่มแชร์

การเปิด Location Services ของมือถืออย่างเดียวไม่พอ ระบบต้องให้คุณกด Start sharing ในหน้า /trip-01 ก่อน จึงจะเริ่ม capture และส่งตำแหน่งออกไป

  • เปิด Location Services/GPS ของมือถือก่อนออกเดินทาง
  • เปิดหน้า /trip-01 ผ่าน HTTPS แล้วกรอกรหัสหน้า trip
  • กด Start sharing ใน Live Location panel
  • เมื่อ browser ขอ permission ให้กดอนุญาตตำแหน่งแบบ “While using this site/app” หรือเทียบเท่า
  • เสียบ power bank/สายชาร์จ และเปิดหน้าไว้ถ้าต้องการให้ update น่าเชื่อถือที่สุด
  • กด Manual ping เมื่อถึงปั๊ม จุดพัก หรือมีเหตุฉุกเฉิน เพื่อส่งตำแหน่งทันทีนอกเหนือจากรอบ 5–10 นาที
  • เมื่อจบทริปหรือไม่อยากแชร์แล้ว กด Stop sharing เพื่อหยุด timer และ revoke/stop session
Viewer state เงื่อนไข สิ่งที่ viewer เห็น สิ่งที่ต้อง implement
Loading กำลังตรวจ token / fetch latest Skeleton หรือข้อความ “กำลังโหลดตำแหน่งล่าสุด” Abort/retry ได้ และไม่โชว์ error หลอกก่อน request จบ
Invalid / expired token token ผิด, หมดอายุ, ถูก revoke ข้อความ “ลิงก์นี้ใช้ไม่ได้หรือหมดอายุแล้ว” ไม่มีพิกัด API คืน 401/403 และ UI ไม่ retry รัว
Waiting first GPS session active แต่ยังไม่มี latest point ข้อความ “เปิดแชร์แล้ว รอตำแหน่งแรกจากมือถือ” poll ต่อทุก 30–60 วินาทีจนมี point หรือ session stopped
Fresh latest age ≤ 15 นาที badge สีเขียว, เวลาล่าสุด, พิกัด, accuracy, ปุ่ม Google Maps คำนวณ age จาก server/client timestamp และ timezone-safe display
Stale latest age > 15 นาที และ ≤ 30 นาที badge สีเหลือง “ข้อมูลเริ่มเก่า” พร้อมเวลาล่าสุดชัดเจน ไม่แสดงเหมือน realtime; เพิ่มคำเตือนว่าอาจยังเดินทางอยู่แต่ยังไม่อัปเดต
Offline / not updating latest age > 30 นาที หรือ owner offline badge สีแดง/เทา “ไม่ได้อัปเดต” และเก็บจุดล่าสุดไว้เป็น last known location แยก last known location จาก current location ใน copy/UI
Stopped owner กด Stop sharing หรือ session จบทริป ข้อความ “หยุดแชร์แล้ว” พร้อม last known location ถ้ายังอนุญาตให้แสดง หยุด polling หรือเปลี่ยนเป็น refresh manual เพื่อลด request
Owner GPS UX รายละเอียดที่ต้องมีใน Live Location panel Error / fallback ที่ต้องแสดง
ก่อนเริ่มแชร์ Checklist: เปิด GPS มือถือ, เปิดหน้านี้ผ่าน HTTPS, เตรียม power bank, อธิบายว่าเว็บอาจหยุดเมื่อ lock screen ถ้า browser ไม่รองรับ Geolocation ให้ขึ้นว่า device/browser นี้ใช้ live location ไม่ได้
Permission prompt กด Start แล้วค่อยเรียก getCurrentPosition() เพื่อให้ browser prompt หลัง user gesture ถ้า denied ให้บอกวิธีไปเปิด permission ใน browser/site settings และให้ปุ่ม retry
กำลังแชร์ แสดง mode, next send countdown, last sent time, accuracy, upload status, wake lock status, viewer link/copy link ถ้า upload fail ให้ retry และบอกว่า viewer ยังเห็น last known location
ระหว่างพัก มีปุ่ม Rest mode เพื่อเปลี่ยน cadence เป็น 10–15 นาที และ Manual ping เมื่อถึงปั๊ม ถ้า accuracy แย่ให้แสดง “ตำแหน่งคร่าว ๆ” แทนการ claim ว่าแม่น
จบทริป ปุ่ม Stop sharing ต้องชัด, มี confirmation สั้น ๆ, แล้วแสดงสถานะหยุดแชร์สำเร็จ ถ้า stop API fail ให้หยุด local timer ก่อนและ retry revoke จนสำเร็จ/แจ้งเตือน

08 · Implementation phases

ลำดับงานที่ต้องทำ

Branch setup & feature flag P0

เริ่มงานบน feat/gps, เพิ่ม feature flag เพื่อให้ GPS panel ปิดได้ถ้า env ยังไม่พร้อม

  • สร้าง/switch branch feat/gps
  • เพิ่ม config helper เช่น lib/trip-gps/config.ts
  • กำหนด default เป็น disabled เมื่อไม่มี env หรือไม่มี token

Tracker panel บน /trip-01 P0

เพิ่ม UI หลัง password gate เพื่อให้ rider เริ่ม/หยุดแชร์ตำแหน่งเองได้

  • แยก component เช่น app/trip-01/live-tracker.tsx
  • ปุ่ม Start sharing, Stop, Manual ping, Battery saver, Rest mode
  • แสดง owner GPS checklist ก่อนเริ่ม: เปิด Location Services/GPS, เปิดผ่าน HTTPS, เตรียม power bank, อย่าล็อกจอถ้าต้องการ update น่าเชื่อถือ
  • เรียก permission หลัง user กด Start เท่านั้น และแสดงวิธีแก้เมื่อ permission denied
  • แสดง permission, current mode, next send countdown, last sent, accuracy, upload status, wake lock status, viewer link/copy link, error
  • ใช้ getCurrentPosition() แบบ periodic snapshot: active 5 นาที, saver 10 นาที, rest 15 นาที
  • ขอ wake lock แบบ optional และแสดง warning เมื่อ page hidden หรือ screen lock อาจทำให้หยุดอัปเดต

Location domain utilities P0

แยก logic ที่ test/verify ได้ออกจาก UI

  • lib/trip-gps/types.ts: payload/session/viewer types
  • lib/trip-gps/cadence.ts: interval per mode, stale/offline calculation
  • lib/trip-gps/geo.ts: accuracy validation, haversine optional, coordinate sanitization
  • lib/trip-gps/client.ts: upload function + retry/offline queue skeleton

API route สำหรับ ingest/latest P1

สร้าง backend boundary ใน Next App Router ก่อนผูก provider จริง

  • app/api/trips/trip-01/location/route.ts รองรับ POST owner upload และ GET viewer latest
  • ตรวจ token, session active, expiry, rate limit, payload shape
  • ตอบ structured errors: 400 invalid payload, 401 invalid token, 403 inactive/revoked, 429 too frequent
  • เริ่มด้วย in-memory/mock adapter เฉพาะ dev ถ้า database credential ยังไม่มี

Storage provider integration P1

ผูก Supabase Postgres เป็น provider หลัก เพราะตรงกับ relational session/latest/history model; Firebase เก็บเป็น backup option เฉพาะกรณีเปลี่ยนโจทย์ไปเน้น managed realtime listener

  • สร้าง adapter interface เช่น LocationStore
  • Supabase tables: trip_share_sessions, trip_location_latest, trip_location_points
  • เปิด RLS หรือบังคับอ่าน/เขียนผ่าน server route เท่านั้น
  • เพิ่ม cleanup/retention script หรือ documented SQL สำหรับลบ history หลัง 24–72 ชม.

Viewer page P1

ทำหน้าสำหรับคนรู้จักดูตำแหน่งล่าสุดแบบ read-only

  • เพิ่ม app/trip-01/live/page.tsx และ client component สำหรับ polling/realtime
  • รับ viewer token จาก query param หรือ path token และตรวจผ่าน API ก่อนแสดงข้อมูล
  • รองรับ states: invalid/expired token, waiting first GPS, fresh, stale, offline, stopped
  • แสดง latest point, timestamp, age, stale/offline state, accuracy, speed optional และข้อความ “last known location” เมื่อข้อมูลเก่า
  • แสดง route stops เดิม + ปุ่มเปิด Google Maps ไปยังจุดล่าสุด + ปุ่ม refresh manual
  • poll ทุก 30–60 วินาที และหยุด/ลด polling เมื่อ session stopped หรือ token invalid
  • ถ้าเลือก map dependency แล้ว ค่อยเพิ่ม marker/accuracy circle/polyline

Cloudflare edge protection + analytics P1

ตั้งค่า Cloudflare ที่ dashboard/domain level ให้รองรับ GPS feature โดยไม่เพิ่ม backend server แยก

  • เปิด DNS proxy/orange cloud สำหรับ hostname ที่ serve เว็บจริง และยืนยันว่า Vercel origin ยังตอบ HTTPS ได้
  • สร้าง WAF/rate limiting rules สำหรับ /api/trips/trip-01/location* และ session endpoints
  • ตั้ง cache bypass/no-store สำหรับ location API และ viewer latest response
  • เพิ่ม Web Analytics เฉพาะ non-sensitive pages; ไม่ฝังบน viewer URL ที่มี token/query
  • บันทึก rule names/thresholds ใน deployment notes เพื่อ rollback/debug ได้

Security & token lifecycle P1

ทำให้แชร์ได้เฉพาะคนที่มีลิงก์ และหยุดแชร์ได้ทันที

  • owner token ส่งตำแหน่งได้เท่านั้น; viewer token อ่านได้เท่านั้น
  • เก็บ token เป็น hash ฝั่ง server/database ไม่เก็บ plain token
  • session มี expires_at, revoked_at, stopped_at
  • Stop sharing ต้อง revoke/mark inactive และ viewer เห็นสถานะหยุดแชร์

Reliability for real trip P2

เพิ่มความทนทานก่อนใช้งานจริงบนถนนยาว

  • offline queue เก็บ point ล่าสุด 1–3 จุดใน memory/localStorage แล้ว flush เมื่อ online
  • UI เตือนเมื่อ page hidden, permission denied, accuracy แย่, upload fail ต่อเนื่อง
  • mode shortcut: Active / Saver / Rest / City approach
  • audit แบบเบา: last viewer access, upload count, last error

Native tracker spike Future

ทำเฉพาะถ้า Web MVP พิสูจน์แล้วว่ายังไม่พอ เพราะต้องการส่งตำแหน่งตอน lock screen จริง

  • ประเมิน Capacitor/React Native companion app
  • ใช้ endpoint/API เดิมของ viewer ต่อได้
  • เพิ่ม background location permission และ platform policy handling

11 · Acceptance criteria

เกณฑ์รับงานที่ทดสอบได้

Functional

  • บน branch feat/gps มี tracker panel ใน /trip-01 หลังปลดรหัสเท่านั้น
  • กด Start แล้ว browser ขอ permission และส่งตำแหน่งแรกทันทีเมื่ออนุญาต
  • active mode schedule ถัดไปทุก 5 นาที; saver mode ทุก 10 นาที; manual ping ส่งทันที
  • กด Stop แล้วหยุด timer, clear watch/lock ที่มี, และ mark session inactive/revoked
  • viewer token valid อ่าน latest location ได้; invalid token อ่านไม่ได้
  • viewer แสดง states ครบ: loading, invalid/expired, waiting first GPS, fresh, stale, offline, stopped
  • viewer แสดง stale เมื่อ latest เกิน 15 นาที และ offline เมื่อเกิน 30 นาที
  • owner panel มี checklist เปิด GPS/Location Services, permission retry, viewer link copy, Start/Stop/Manual ping/Rest mode
  • owner กด Start เท่านั้นจึงเรียก permission และเริ่มส่งตำแหน่ง; ก่อนกด Start ไม่มีการส่ง location ออกไป
  • Cloudflare DNS proxy เปิดสำหรับ hostname เว็บจริง และ API location ไม่ถูก cache
  • Cloudflare WAF/rate limiting ไม่บล็อก viewer polling ปกติ แต่ challenge/block request burst ที่ผิดปกติ
  • Web Analytics ไม่ถูกฝังบน viewer URL ที่มี token/query หรือมีแผน sanitize URL ก่อน track

Quality / security

  • ไม่มี server secret/service role key ใน client bundle หรือ NEXT_PUBLIC_*
  • payload validation ปฏิเสธ lat/lng นอก range, accuracy ติดลบ, timestamp invalid
  • rate limit ป้องกัน upload ถี่ผิดปกติ เช่นถี่กว่า 30 วินาที ยกเว้น manual/debug ที่กำหนด
  • history retention ถูก document หรือมี cleanup job/script
  • npm run lint และ npm run build ผ่านก่อนจบงาน
  • ถ้าไม่สามารถทดสอบบนมือถือจริงได้ ต้องระบุ validation gap ชัดเจน

12 · Verification plan

วิธีตรวจงานก่อน merge

A

Static verification

  • npm run lint
  • npm run build
  • ตรวจว่า route handlers ไม่ cache latest location ผิดพลาด
  • ตรวจว่า env names แยก public/server-only ถูกต้อง
  • ตรวจ response headers/API behavior ว่า location API เป็น dynamic/no-store และไม่ถูก Cloudflare cache
  • ตรวจ Cloudflare rule preview/logs ว่า WAF/rate limiting match เฉพาะ endpoint ที่ตั้งใจ
B

Browser/device verification

  • ทดสอบ Chrome/Safari บนมือถือจริงผ่าน HTTPS preview
  • permission allow/deny/error path และวิธี retry หลังเปิด permission ใหม่
  • owner flow: ก่อน Start ไม่ส่งตำแหน่ง, หลัง Start ส่งทันที, Manual ping ส่งทันที, Stop หยุด sharing
  • viewer flow: valid token เห็นข้อมูล, invalid/expired token ไม่เห็นพิกัด, waiting/fresh/stale/offline/stopped แสดงถูกต้อง
  • page hidden warning และ wake lock fallback
  • ปิดเน็ตแล้วกลับมา online เพื่อดู retry/offline queue
C

Security verification

  • owner token ใช้ GET viewer ไม่ได้ถ้าแยก policy ไว้
  • viewer token POST ไม่ได้
  • expired/revoked token ถูก reject
  • Stop sharing ทำให้ viewer เห็น stopped state
  • ทดสอบว่า viewer token ไม่ถูกส่งเข้า analytics beacon/network request ที่ไม่จำเป็น
